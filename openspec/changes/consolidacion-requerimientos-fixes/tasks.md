- **Type:** implementation
- **Sequential after:** nothing (first task)
- **Spec requirement:** Item Lock on Consolidation (en_consolidacion); Consolidation Event in Request History
- **File:** `supabase/migrations/024_consolidacion_fixes.sql` (NEW)
- **What:**
  - `BEGIN/COMMIT` block.
  - `ALTER TABLE request_items DROP CONSTRAINT IF EXISTS chk_item_status; ADD CONSTRAINT chk_item_status CHECK (status IN ('sin_pedir','en_oc','parcial','recibido','en_consolidacion'))`.
  - `ALTER TABLE requerimiento_evento DROP CONSTRAINT IF EXISTS chk_evento_tipo; ADD CONSTRAINT chk_evento_tipo CHECK (tipo IN (...12 values...,'consolidado'))`.
  - `CREATE OR REPLACE FUNCTION create_consolidated_rfq(p_company_id uuid, p_created_by uuid, p_lines jsonb) RETURNS uuid LANGUAGE plpgsql SECURITY INVOKER` with the full body: INSERT rfqs → rfq_items → rfq_item_sources → rfq_requests → UPDATE request_items (lock, guarded by `status='sin_pedir'`) → INSERT requerimiento_evento DISTINCT by request_id → RETURN rfq_id.
  - Rollback block commented at the bottom (normalizes data before restoring strict sets).
- **Manual verification checklist (record in PR):**
  - INSERT `request_items` with `status='en_consolidacion'` passes CHECK.
  - INSERT `requerimiento_evento` with `tipo='consolidado'` passes CHECK.
  - `SELECT create_consolidated_rfq(...)` returns a UUID and sets items to `en_consolidacion`.
  - Rollback script restores original 4-value set after normalizing data.
- **Notes:** DROP+ADD takes `ACCESS EXCLUSIVE` briefly; apply in low-traffic window. Superconjunto → no row rewrite.
