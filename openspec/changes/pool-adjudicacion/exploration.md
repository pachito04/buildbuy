# Exploration: Pool de Compras — Adjudicación (#9c)

Source: `Reporte 1805.docx` — Pool de Compras "Comparativa y adjudicación en Pool" + "Trazabilidad". Final piece, builds on `#9a` (links/mappings) + `#9b` (flow: shared RFQ, contributions, states).

## Report requirements (#9c)

- The comparativa is **centralized**; all pool participants can see it.
- Adjudication: a "leader" company adjudicates OR each company adjudicates its own portion.
- Once adjudicated, **each company generates its OWN purchase orders** for the portion that corresponds to it (the provider quote is single; the system splits internally per company).
- Traceability: each pool line shows units per company; OCs are individual per company.
- Confidentiality: each participant sees the consolidated comparativa, not other companies' internal requirement detail (already enforced by `#9b`'s `pool_requests` RLS).

## Current state (evidence)

- **Comparativa** (`Comparativa.tsx`) loads an RFQ + its quotes; already selects `pool_id` + `purchase_pools(name)`. But `rfqs_tenant` RLS (001:784) is `company_id = auth_company_id()` → a **pool RFQ is visible ONLY to the owner company**. `rfq_items_tenant` and `quotes_company_view` are likewise own-RFQ-scoped. ⇒ pool participants cannot see the shared comparativa.
- **Adjudication** (`RFQs.tsx` `awardFromRfq`, ~:158-195): creates ONE `purchase_orders` row for `companyId` with the FULL quote total and all `quote_items` at full quantity. For a pool this is wrong — each company must order only ITS contributed quantity.
- `purchase_orders` INSERT RLS is `company_id = auth_company_id()` → a user can only create POs for their OWN company. So a pool's per-company OCs MUST each be generated by that company (cannot be created in one cross-company write).
- `#9b` produced: the shared RFQ (`rfq_type='pool'`, `pool_id`), `pool_items` (one per material), and `pool_item_contributions` (per-company quantity per pool line). The shared RFQ's `rfq_items` were created from `pool_items` (by material), so rfq_item ↔ pool_item is by material within the pool.

## What #9c must add

1. **RLS widening (additive SELECT policies)** so pool members can read a **pool** RFQ's comparativa: `rfqs`, `rfq_items`, `quotes` (and quote_items) SELECT widened to "viewer is a member of the pool that owns this pool RFQ" (`rfqs.pool_id` → `is_pool_member`). Non-pool RFQs unchanged (own-company only).
2. **Per-company OC generation**: for a pool RFQ, when a winning quote is adjudicated, EACH participating company generates its own `purchase_orders` for its contributed quantity per line (quantity from `pool_item_contributions` for that company; unit_price from the winning quote_item). Each company runs its own "generar mi OC" (RLS-enforced own-company).
3. **Pool states**: `adjudicado` (a winner chosen) → `cerrado` (all participants generated their OCs).
4. Pool line per-company breakdown surfaced in the shared comparativa (reuse `pool_item_contributions`).
